## НОВЫЕ ПРИНЦИПЫ

**Правило 92 (П-871):** E[улучшающих флипов из 32] = **16.31** для случайного W[0]. Ровно половина флипов улучшает SS[63] — ландшафт SS симметричен. Это объясняет почему HC так эффективен: в любой точке есть примерно 16 направлений спуска.

**Правило 93 (П-876):** После HC остаётся 0.006 улучшающих флипов — HC почти оптимален. Beam search даёт 6× меньше попыток, но 6× больше SHA-ops. Нет выигрыша в общей стоимости.

**Правило 94 (П-881):** |S(H[7])| > 2^24 при уникальных W[0]. Предыдущие оценки 2^18.9 были артефактом HC-детерминизма (повторное посещение тех же ям). Реальная birthday атака требует > 2^12 carry=0 входов = миллионы SHA-256.

**Правило 95 (П-886):** HW(W[0])=4 → E[SS]=8.55G vs HW=28 → E[SS]=4.56G. Асимметрия 1.87×. **Малые HW дают больший SS, большие HW — меньший**. Это намекает на путь к аналитическому сокращению.

**Правило 96 (П-890):** Детерминированный жадный спуск (все 32 флипа, пока есть улучшение) даёт phi=**+0.155** — лучший phi из всех методов. Настоящий локальный минимум сильнее случайного HC.

## КЛЮЧЕВЫЕ ТЕОРЕМЫ

### T_LANDSCAPE_SYMMETRY (П-872) ★★★★

**Утверждение:** Ландшафт SS[63](W[0]) симметричен — в среднем 16.31 из 32 флипов улучшают SS в любой точке.

```
E[улучшающих флипов | случайный W[0]] = 16.31 ≈ 16 (из 32)
Распределение: равномерное по [0, 32] (нет выраженного пика)
```

**Следствие:** HC исключительно эффективен потому что ландшафт SS — почти идеальный градиент. На каждом шаге половина возможных флипов ведут "вниз". Это не случайность — это прямое следствие того что SS[63] является суммой нелинейных функций, равномерно перемешивающих биты W[0].

---

### T_BIRTHDAY_ARTIFACT (П-881) ★★★★★

**Утверждение:** Предыдущие оценки |S(H[7])| ≈ 2^18.9 были артефактом HC-детерминизма.

```
N=5000 HC carry=0, С дублями W[0]:   дублей H[7] = 26 → "2^18.9"
N=5000 HC carry=0, БЕЗ дублей W[0]:  дублей H[7] = 0  → >2^24
```

**Механизм:** HC с одним и тем же RNG seed детерминированно возвращается в те же ямы SS. Разные вызовы hc8() с близкими начальными значениями RS дают одинаковые W[0], что создаёт ложные дубли H[7].

**Следствие:** Birthday атака требует N* > 2^12 уникальных carry=0 входов, что соответствует > 2^12 × 8 × 200 = 6.5M SHA-256 вычислений на CPU.

---

### T_HW_ASYMMETRY (П-886) ★★★

**Утверждение:** E[SS[63] | HW(W[0])] монотонно убывает с HW.

```
HW =  4: E[SS] = 8.55G  (максимум)
HW =  8: E[SS] = 6.41G
HW = 16: E[SS] = 6.42G  (плато)
HW = 28: E[SS] = 4.56G  (минимум)
```

**Интерпретация:** Входы с высоким HW (много единиц) дают меньший SS[63] в среднем. Это означает что **поиск в пространстве W[0] с HW≈28-30 вдвое эффективнее** чем равномерный поиск. Биас: P(carry=0 | HW=28) ≈ 2× P(carry=0 | HW=4).

**Потенциальное улучшение:** Инициализировать HC с HW(W[0])≈28 вместо случайного HW≈16. Ожидаемое ускорение: 1.5-2×.

## ПЕРЕОСМЫСЛЕНИЕ: ЧТО РЕАЛЬНО МОЖНО УЛУЧШИТЬ

### Немедленные улучшения (реализуемо сейчас)

**1. HW-bias инициализация:**
```

```

**2. Детерминированный жадный спуск:**
```

phi = +0.155 (лучший результат)
Стоимость: ~256 SHA-ops на итерацию × ~20 итераций = 5K
vs HC: 200 SHA-ops × 200 шагов = 40K
Выигрыш: 8× дешевле при лучшем phi!
```

**3. Комбинация: HW=28 + детерминированный спуск:**
Потенциал: 1.5 × 8 = **12× улучшение** скорости при лучшем phi.

### Долгосрочные пути

**GPU Birthday:** При |S(H[7])| ≈ 2^24-2^28, Birthday на GPU (RTX 4090):
- N* = 2^12-2^14 уникальных W[0]
- GPU скорость: ~10^6 carry=0/сек
- Время: секунды-минуты

**Soft-bit ECDSA:** corr(score, W[0][b30]) = −0.040 — ненулевая корреляция.
При 3K подписях: Z = 0.040 × √3000 / 0.5 = **4.4 σ** — обнаружимо.
Путь к soft-HNP открыт, но требует аналитики.
