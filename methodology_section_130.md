## ПУТЬ 1: БАЙЕСОВСКИЙ PREIMAGE

**Идея:** знание H → вывод о carry[63]=0 → сужение пространства W.

**Данные:**
- score(H) > −0.246 → P(carry=0 | H) ≈ 71%
- E[HW(W[0]) | carry=0] = 16.02 vs 16.00 случайного
- **HW(W[0]) НЕ смещён** при carry=0 — нет информации о W[0]

**Вывод:** Байесовский preimage не работает.
Знание carry[63]=0 не раскрывает ничего о конкретном W[0].
Ямы SS равномерно распределены по HW(W[0]).

## ПУТЬ 2: ЧАСТИЧНАЯ КОЛЛИЗИЯ H[7]∧H[6]

### Ключевое открытие: H[6] функционально зависит от H[7]

**Данные (N=10000, K=8):**
- Дублей H[7]: 127 → при HC-процедуре
- Дублей H[6]: 127 → **то же самое множество пар**
- При коллизии H[7]: H[6] совпадает в 100% случаев (3/3)

**Интерпретация:**
```
corr(H[7], H[6]) = −0.043 (линейно независимы!)
НО: при H[7]_a = H[7]_b → H[6]_a = H[6]_b всегда.
```
Это **нелинейная функциональная зависимость**:
H[6] = f(H[7]) при carry[63]=0. Знание H[7] однозначно
определяет H[6] в carry=0 режиме.

**Причина:** H[7]=g[62]+IV[7], H[6]=f[62]+IV[6].
При carry[63]=0 конкретные значения g[62] и f[62]
жёстко связаны через финальные раунды SHA-256.

### Противоречие с birthday атакой

| Метод оценки |  |S(H[7])| |
|---|---|
| N=5000 HC дублей | ≈ 2^18.9 |
| N=10000 HC дублей | ≈ 2^18.6 |
| Birthday (21K без коллизии) | > 2^28.2 |

**Разгадка:** дубли H[7] при HC — **артефакт детерминизма HC**.
HC повторно попадает в одни и те же W[0] при схожих seed.
Реальный |S(H[7])| >> 2^19.

**T_HC_DUPLICATES_ARTIFACT (П-841):** Дубли H[7] в HC-банке
не отражают реальный размер образа. HC-процедура детерминированно
возвращается в одни ямы при схожих начальных условиях.
Реальный |S| оценивается через birthday — > 2^28.

**Следствие для коллизии:**
```
|S(H[7])| > 2^28
Birthday N* > √(2^28) = 2^14 = 16384 carry=0 входов
= 16384 × 8 × 200 = 26M SHA-256 вычислений
```
Это **практически недостижимо** на одном CPU.

## ПУТЬ 3: SCORE → HNP KEY RECOVERY (ECDSA)

**Энтропия score v4.1 при carry=0:** ~3.35 бит

**Теоретическая утечка на подпись:**
- Через 1 бит H[6][b29]: 0.144 бит → ~1778 подписей
- Через score v4.1: если score = f(bits(k)), то ~77 подписей

**Проблема:** score — нелинейная функция H, а H — нелинейная
функция nonce k. Прямой связи score ↔ bits(k) нет.

**Что нужно для улучшения:**
Построить soft-bit предсказатель: P(k[b]=1 | score, msg).
Это возможно но требует отдельного исследования.

## ЧЕСТНАЯ ОЦЕНКА: ЧТО РЕАЛЬНО УЛУЧШАЕТ АТАКУ

### То что работает прямо сейчас

```
1. Distinguisher v4.1:
   AUC=0.765, Advantage=+0.529, Z=49.7
   За 814 SHA-256 → обнаружение SHA-256 с P>0.9999

2. Commitment hiding (R114):
   Advantage=+0.300 → hiding нарушена при chosen-r

3. ECDSA без RFC 6979 (R115):
   ~1778 подписей → приватный ключ

4. Merkle tree (R115):
   Advantage=+0.131
```

### Что нужно для настоящей коллизии

```
Нужно: |S(H[7..0])| << 2^128
Имеем: |S(H[7])| > 2^28 (только один выходной слой)
Разрыв: ~100 бит

Единственный реалистичный путь:
  Найти алгебраическое ограничение на H[7]
  которое сокращает |S| с 2^28 до 2^14.
  Это потребует новых математических инструментов.
```

### Главные мысли для следующего шага

**Идея 1 — Soft-bit ECDSA:**
Если score v4.1 несёт 3.35 бит информации о хеше,
и если эта информация частично "течёт" в bits(k) nonce,
то можно построить soft-HNP редукцию с LLL.
Потенциал: 77 подписей вместо 1778.

**Идея 2 — Многоблочное сообщение:**
SHA-256 многоблочного сообщения — IV второго блока = H от первого.
При carry[63]=0 в блоке 1 → IV второго блока смещён.
Phi(H2) ≈ +0.057 (из раздела 127).
Итерация 3+ блоков — phi убывает экспоненциально.

**Идея 3 — GPU параллелизация:**
Birthday за 2^14 входов при K=64:
= 16384 × 37 = 606K SHA-256
На GPU (10^9 SHA/сек) = 0.6 мс.
**Birthday коллизия H[7] на GPU — практически мгновенно.**

**Идея 4 — Алгебраический анализ carry:**
carry[63]=0 — линейное условие на SS63+K[63]+W[63] < 2^32.
SS63 — нелинейная функция W[0] через 63 раунда.
Задача: найти алгебраические корни SS63 + C < 2^32.
Это SAT/SMT задача. При правильном кодировании — решаема?
