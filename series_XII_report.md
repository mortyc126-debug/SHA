## 2. ДИАГНОЗ: ПОЧЕМУ PHASE 2/3 НЕ ПОМОГАЕТ

**T_PHASE_BARRIER (критический диагноз):**

Все рекорды (`hw59=83,84,85,86`) получены в фазе **PDESCENT** (Phase 1), а не PREF INE или PAGGINV. Это означает: переключение на inv-push при hw59<95 **мешает** дальнейшему снижению hw59.

```
Phase 1 objective: f = hw59
Phase 2 objective: f = hw59 + 15*(6-inv)/6

Если inv=3/6 (типично):
  Phase 2 добавляет штраф +7.5 к objective
  → SA принимает мутации, которые поднимают hw59 ради улучшения inv
  → hw59 плато застревает выше, чем мог бы
```

**Вывод:** hw59 и inv=6/6 — конкурирующие цели в данном ландшафте. Composite objective тянет в обе стороны одновременно, что хуже, чем чистая минимизация hw59.

### Intersection probe (50 trials × 300K шагов)

```
P(hw59<90)           = 0/50 = 0.000
P(inv=6/6)           = 12/50 = 0.240  ← inv=6/6 легко достижимо!
P(hw59<90 AND inv=6) = 0/50 = 0.000   ← пересечение не наблюдалось
```

**T_PROBE_EMPTY:** при 300K шагов ни разу не наблюдалось hw59<90 AND inv=6/6 одновременно.

**T_INV6_INDEPENDENT:** inv=6/6 достигается с P=0.24 — это часто! Но при hw59=106..120, не при <90. Это разные области W-пространства.

**Структурная гипотеза T_INV_HW_ANTI:**
```

  Эти же паттерны могут УВЕЛИЧИВАТЬ дифференциал в раундах 17..59.

То есть: W, хороший для Zone-3, может быть плохим для Zone-2.
```

## 4. НОВАЯ СТРАТЕГИЯ: SEQUENTIAL ATTACK

**Идея (Series XIII):**

Вместо одновременной оптимизации hw59 И inv — двухэтапная последовательность:

```
Stage 1: Pure DESCENT
  Найти W* с hw59 = минимум (цель: <80)
  Не заботиться об inv вообще.

Stage 2: Local Inv Search
  Из найденного W*: варьировать ТОЛЬКО W[48..60] (schedule zone-3)
  W[0..47] — фиксированы.
  Искать вариацию δW[48..60] такую, что:
    - hw59 изменяется минимально (≤2 бита)
    - inv улучшается до 6/6

Почему это может работать:
  W[48..63] через schedule влияют ТОЛЬКО на W[48..63] раунды.
  Изменение W[48..60] меняет состояние с r=48 — это 11 раундов до r=59.
  hw59 может быть нечувствителен к малым вариациям W[48..60].
```

**Дополнение: Schedule Decoupling Lemma**
```
W[i] для i=0..15 задаются свободно.
W[i] для i=16..63 жёстко определены через s0/s1.
НО: W[48..60] зависят от W[32..58] — то есть от W[0..15] через 
    промежуточные ступени расписания.

Это означает: варьировать W[48..60] напрямую невозможно без
изменения W[0..47]. Однако мы можем варьировать W[k] для k∈{0..15}
с условием, что δW[0..47] ≈ 0 (т.е. изменение только в W[48..63]).

Из schedule: W[48] = s1(W[46]) + W[41] + s0(W[33]) + W[32]
  → зависит от W[0..15] через 32..46 ступени.
  → "локальный" контроль W[48] через W[k] минимально затрагивает W[16..47].
```

## 5. НОВЫЕ ТЕХНИКИ, ПОКАЗАВШИЕ ПОТЕНЦИАЛ

### Gene Transplant (качественный сигнал)

При crossover из pool: некоторые прогоны достигали hw59=84 быстрее (за 7M итераций vs 12M без crossover). Небольшое ускорение — пул помогает, но не критически.

### Zone3-Pull operator

```mutate_zone3_pull``` мутирует W[13..15], которые через schedule влияют на W[45..63]. Наблюдение: этот оператор улучшал inv, но поднимал hw59 на 2-3 бита. Подтверждает T_INV_HW_ANTI.
