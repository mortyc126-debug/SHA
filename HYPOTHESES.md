## H1 — p-адическая башня SHA-256 (★★★ ПРИОРИТЕТ)

**Источник**: П-49..П-53 (T_SHA_PADIC_TOWER, T_CASCADE_UNIQUENESS, T_INFINITE_TOWER)

**Формулировка**:
```
Определим Sol_k = {DW ∈ (Z/2^k)^16 : F(DW) ≡ 0 mod 2^k}
где F(DW) = (De3, De4, ..., De17) — дифференциальная система.

height_2(SHA-256) = sup{k : Sol_k ≠ ∅}

Известно (П-53): height_2 ≥ 11.
Гипотеза T_INFINITE_TOWER: height_2 = ∞ (башня бесконечна).
```

**Что проверяем**:
1. P(Sol_1 ≠ ∅) исчерпывающим перебором в {0,1}^15 (≠ greedy): П-57 → 75% (birthday slice)
2. Можно ли поднять Sol_k → Sol_{k+1} для k > 11?
3. Есть ли k где подъём блокируется (T_MOD4_BARRIER_ABSOLUTE для k=2)?

> **П-57 UPDATE**: "9% феномен" — артефакт жадного алгоритма, не свойство SHA-256.
> P(Sol_1, exhaustive) = 75% (birthday slice), greedy даёт лишь 0.6-9% из них.
> SHA-256 ведёт себя как случайная функция (88% vs 86% random в {0,1}^16).

**GPU эксперимент** (`p_adic_tower`):
```bash
./p_adic_tower 10000000 30 tower_h1_deep.txt
```

**Ожидаемые результаты**:
- P(Sol_1, greedy) ≈ 9% — артефакт алгоритма; P(Sol_1, exhaustive) = 75% (П-57)
- height_2 ≥ 20 → башня растёт, T_INFINITE_TOWER вероятна
- height_2 = k* < ∞ → точный барьер, связан с v₂(σ₁(1)) = 13

**Значимость**: Если height_2 = ∞, то SHA-256 имеет принципиально новое
алгебраическое свойство. Если finite: точное значение k* = новый инвариант.

## H2 — Статистика Da13 на 10k+ парах (★★★)

**Источник**: П-32 (T_STATE17), П-37 (A3 отрицательный для 3 пар)

**Формулировка**:
```
При De3..De17=0, Da13 = a13(W') - a13(W).
T_STATE17: Da13 — случайное 32-битное число (HW ≈ 16, насыщение).
T_BARRIER17_EXACT: Da13 + DW16 = 0  (500/500 верифицировано).

Гипотеза H2a: HW(Da13) строго равномерно на [1..31]?
Гипотеза H2b: Есть ли корреляция между W0 и Da13?
Гипотеза H2c: Существуют ли "удачные" W0, дающие P(f17=0) > 2^{-32}?
```

**GPU эксперимент**:
```bash
./birthday_search_17 0 1 100 results_da13.csv   # 100 random W0 → ~100 pairs
python3 analyze_da13.py results_da13.csv          # статистика
```

**Код анализа** (`analyze_da13.py`):
```python
import sys, collections
pairs = [line.split(',') for line in open(sys.argv[1]) if line.startswith('0x')]
da13s = [int(p[3],16) for p in pairs if len(p)>=5]
hws   = [bin(x).count('1') for x in da13s]
print(f"N={len(da13s)}, HW: min={min(hws)} max={max(hws)} mean={sum(hws)/len(hws):.3f}")
# Check bit distribution
for bit in range(32):
    frac = sum((x>>bit)&1 for x in da13s)/len(da13s)
    if abs(frac-0.5) > 0.1:
        print(f"BIAS: bit {bit} P={frac:.3f}")
```

**Ожидаемый результат**: HW(Da13) ≈ 16 ± σ, все биты ≈ 50%.
Если HW(Da13) ≠ 16 или есть битовый bias → ускорение birthday до 2^{31}.

## H3 — DW0 = 2^j для j = 0..31: равноправны ли все биты? (★★)

**Источник**: П-37 (C4 — j=24 гипотеза), T_CARRY_GEOMETRY

**Формулировка**:
```
Текущий алгоритм: DW0 = 1 (бит 0 флип).
Гипотеза C4: j=24 даёт E[HW(Da1)] = 1 (меньший вклад в Da13).

Проверяем: P(De17=0) одинаков для всех j=0..31?
Или некоторые j дают P > 2^{-32} → более быстрый birthday?
```

**GPU эксперимент** (встроен в `run_experiments.sh`):
```bash
for j in {0..31}; do
    DW0=$((1 << j))
    ./birthday_search_17 e82222c7 $(printf %x $DW0) 10 results_j${j}.csv
done
```

**Ожидаемый результат**: Для всех j одинаково ~10 пар за 10 × 2^32 попыток.
Если j* даёт больше пар → P(De17=0 | DW0=2^{j*}) > 2^{-32} → прорыв.

**Примечание**: T_DE17_UNIFORM (П-13) говорит о равномерности Da13 при j=0.
Для j ≠ 0 аналогичное утверждение NOT проверено — открытый вопрос.

## H4 — Нейтральные биты в Sol_2 → ускорение до Sol_3 (★★★)

**Источник**: П-53 (T_CASCADE_UNIQUENESS, нейтральные биты), T_NEUTRAL_STEP (П-12)

**Формулировка**:
```
Известно: для De_{3k}=0 существуют нейтральные биты (W[1]: бит 15, W[4]: {5,6,7}, etc.)
Аналог: если для Sol_2 существует бит b такой, что
  DW XOR 2^b ∈ Sol_2  для любого DW ∈ Sol_2
то |Sol_2| ≥ 2 → легче найти Sol_3.

Гипотеза H4: Sol_2 имеет нейтральные биты (структура Z_2^m).
```

**Проверка**:
```python
# После нахождения DW* ∈ Sol_2 (через p_adic_tower):
for b in range(16*32):  # все 512 битов в DW[0..15]
    DW_mod = DW.copy()
    coord, bit = b // 32, b % 32
    DW_mod[coord] ^= (1 << bit)
    if check_sol2(DW_mod):  # F(DW_mod) ≡ 0 mod 4
        print(f"Нейтральный бит: координата {coord}, бит {bit}")
```

**Значимость**: Если m нейтральных битов → |Sol_2| ≥ 2^m →
вероятность нахождения Sol_3 при случайном подъёме = 2^{-15+m} вместо 2^{-15}.

## H6 — Исчерпывающий поиск Sol_3 через GPU (★★★)

**Источник**: П-48 (0/393K кандидатов mod 4 → барьер?), П-51 (жадный mod-8 каскад)

**Формулировка**:
```
(Z/4Z)^16 имеет 4^16 = 2^32 элементов.
Проверить ВСЕ → слишком долго (CPU: дни).
GPU: при 110G SHA/s → 2^32 SHA за 0.04 сек!

Гипотеза: Sol_3 ≠ ∅ (П-51 показал Sol_11 ≠ ∅ через жадный каскад).
Вопрос: СКОЛЬКО элементов в Sol_3?

GPU эксперимент: перебор DW ∈ {0,1,2,3}^16 параллельно.
Cost: 4^16 = 2^32 SHA → 0.04 sec на A100!
```

**GPU код**:
```cuda
__global__ void exhaustive_sol3(uint32_t W0, uint32_t* d_count) {
    uint64_t idx = (uint64_t)blockIdx.x * blockDim.x + threadIdx.x;
    if (idx >= (1ULL<<32)) return;  // 4^16 = 2^32

    uint32_t DW[16];
    for (int i = 0; i < 16; i++)
        DW[i] = (idx >> (2*i)) & 3;  // each coord in {0,1,2,3}

    // Check F(DW) ≡ 0 mod 8
    if (check_differential_mod8(W0, DW))
        atomicAdd(d_count, 1);
}
```

**Значимость**: |Sol_3| = ? даёт точное значение height_2 в контексте этого W0.

### ❌ H1 — p-адическая башня — БАГ В КОДЕ

**Обнаруженная ошибка** в `p_adic_tower.cu` (строки 81-82):
```c
// БЫЛО (неправильно):
#define sig0(x)  (ROTR32(x,7)^ROTR32(x,18)|((x)>>3))   // | вместо ^
#define sig1(x)  (ROTR32(x,17)^ROTR32(x,19)|((x)>>10))  // | вместо ^
```
Приоритет операторов C: `^` > `|`, поэтому `a^b|c` = `(a^b)|c` ≠ `a^b^c`.
Это ломает вычисление W[16] в расписании → результат 0.003% вместо ожидаемых 9% Sol_1.

**Исправление применено**: `|` заменены на `^` в обоих макросах (исправлено 13 марта 2026).

**Требует перезапуска**:
```bash
make ARCH=sm_80
./p_adic_tower 10000000 30 tower_h1_fixed.txt
```
Ожидаемый результат после исправления: P(Sol_1, greedy) ≈ 9% (алгоритмический артефакт),
P(Sol_1, exhaustive) = 75% (П-57), height_2 ≥ 11.

### ✅ П-57 — T_JACOBIAN_RANK_PREDICTS_SOL1 + феномен 9% объяснён (Март 2026)

**Источник**: П-57, `jacobian_gf2.py`, 1000 сидов

```
Матрица Якоби J_B (15×15) в точке DW=(1,0,...,0) над GF(2):
  rank = 15 у 53% W0_seed  → P(лин. решение) = 100%
  rank = 14 у 47% W0_seed  → P(лин. решение) = 50%
  Rank НИКОГДА не опускается ниже 14 из 1000 сидов.

T_JACOBIAN_RANK_PREDICTS_SOL1 ПОДТВЕРЖДЕНА:
  P(F_ref ∈ Image(J_B)) = 2^{rank(J_B) - 15}
  Теория точно совпадает с экспериментом (rank=15→100%, rank=14→50%).
```

**Феномен 9% — закрыт**:
```
SHA-256 в {0,1}^16:
  P(Sol_1, exhaustive) = 75%  (birthday slice, DW[0]=1)
  P(Sol_1, random fn)  ≈ 63%  ← SHA-256 ЛУЧШЕ random
  P(Sol_1, greedy cascade) = 0.6%  ← алгоритмический артефакт
  Разрыв exhaustive/greedy ≈ 125×

  Full space {0,1}^16:
  P(Sol_1, exhaustive) = 88.2%  vs  86% random  ← SHA-256 ≈ random ✓
```

**Вывод**: 9% — не свойство SHA-256, а низкая эффективность конкретного жадного алгоритма.
